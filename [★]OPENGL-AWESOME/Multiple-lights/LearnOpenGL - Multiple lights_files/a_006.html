                    And here's my take on this, using good old sprintf() function:

                    glm::vec3 colorRed = glm::vec3(1.0f, 0.0f, 0.0f);
                    glm::vec3 colorGreen = glm::vec3(0.0f, 1.0f, 0.0f);
                    glm::vec3 colorBlue = glm::vec3(0.0f, 0.0f, 1.0f);
                    glm::vec3 colorYellow = glm::vec3(1.0f, 1.0f, 0.0f);

                    glm::vec3 pointLightColors[] = {colorRed, colorGreen, colorBlue, colorYellow};

                    for (int i = 0; i < 4; i++)
                    {
                      char buffer[64];

                      sprintf(buffer, "pointLights[%i].position", i);
                      setUniformVec3(lightingShader, buffer, pointLightPositions[i]);

                      sprintf(buffer, "pointLights[%i].constant", i);
                      setUniformFloat(lightingShader, buffer, 1.0f);

                      sprintf(buffer, "pointLights[%i].linear", i);
                      setUniformFloat(lightingShader, buffer, 0.09f);

                      sprintf(buffer, "pointLights[%i].qaudratic", i);
                      setUniformFloat(lightingShader, buffer, 0.032f);

                      sprintf(buffer, "pointLights[%i].diffuse", i);
                      setUniformVec3(lightingShader, buffer, pointLightColors[i]);

                      sprintf(buffer, "pointLights[%i].specular", i);
                      setUniformVec3(lightingShader, buffer, glm::vec3(0.0f));

                      sprintf(buffer, "pointLights[%i].specular", i);
                      setUniformVec3(lightingShader, buffer, glm::vec3(1.0f));
                    }

                                                                                        JoeyDeVries Sensei Guest 6 years ago

                                                                                        You usually define an upper bound (let's call this n) and update the array accordingly, taking the n closest lights at any time and using those n lights to fill the light array. If a light gets too far, it isn't considered anymore for lighting in this array and replaced with a more closer light.

                                                                                        A more flexible approach would be to use deferred rendering with rendering a light volume for each (close enough) light in your scene. With that approach the fragment shader doesn't need to know about the numer of lights, you simply issue a draw call for each light you want to contribute.

                                                                                                                                              JoeyDeVries Sensei Boreas 5 years ago

                                                                                                                                              You're right on all points. The system we have here isn't at all physically accurate as light sources only have one type of light and surfaces can reflect/refract this light in a specular/diffuse way respectively, which should purely be modeled on the surface materials (not the lights). However, this is a fairly simple system to understand, gives users quite a lot of flexibility when it comes to lighting so makes for an easier introduction.

                                                                                                                                              If you want your lighting to follow more physical terms, you'd probably be interested in the PBR tutorials (however these are quite advanced so you'll want some more OpenGL knowledge). Your suggestions would already make for a nice leeway to a more physical based system so definitely go for that :)
